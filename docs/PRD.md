# 今日谕句 PRD

## 1. 项目概述

### 1.1 项目名称

暂定名：今日谕句

### 1.2 一句话定义

一个以内容为核心的情绪陪伴产品。每天给用户一条"谕句"，再给一条"宜"和一条"忌"。谕句偏留白，像《答案之书》那样，让人自己去对照；宜忌不走老黄历，也不追求工整对仗，而是结合天气、地点和情绪，给出更轻一点的提示。

### 1.3 项目目标

先把内容生产链路做通，再做 App 与小组件。

这个项目最难的不是客户端，而是内容系统。我们不是做随机鸡汤，也不是做名言摘抄工具。我们要做的是一条稳定的内容生产线：

- 能上传图书、歌词和其他文本
- 能自动切片、初筛、打标签
- 能逐条审核，是否放弃
- 能把内容分发给每台设备，并尽量避免重复

## 2. 产品定位

### 2.1 目标用户

第一阶段先不把用户画像写得太满，先抓一个比较真实的人群：

- 愿意被一句话打动的人
- 会在某个时刻想看看"今天该怎么过"的人
- 对鸡汤、说教和过度包装有点反感的人
- 不需要社交功能，只想安静地接收一点内容的人

### 2.2 核心产品原则

- 谕句、宜、忌分开建库，不绑定成固定套餐
- 内容必须经过人工终审
- 第一版标签少一点，系统先做轻
- 宜忌不要求对仗，只要求自然
- 不做账号体系，但要支持设备级去重
- App 直连 Supabase（anon key + RLS），不单独搭分发服务

## 3. 内容定义

### 3.1 谕句

谕句是来自书籍/歌词等类型文本的片段，短小，能触发人的情绪和思考，但不需要完整的故事或情节。

### 3.2 宜

宜是轻一点的行动建议，偏顺势，不用太工整。

要求：

- 可以具体一点
- 不要命令口吻太重
- 不追求对仗
- 读起来要自然

### 3.3 忌

忌是提醒，偏避免误判，不要太重，也不要突然变成道德说教。

要求：

- 不要太吓人
- 不要突然变成训诫
- 不要总用很重的词

## 4. 开发阶段规划

项目分两阶段推进。

### Phase 1：内容后台与语料库系统

目标是把"上传文本 -> 自动分析 -> 审核入库"这条链路跑通。

交付内容：

- Web 后台管理界面
- Supabase 数据库
- 文件上传流程
- 文本分析服务
- 审核页
- 正式内容库

阶段结果：

- 建立谕句库
- 建立宜库
- 建立忌库
- 每条内容都经过终审

### Phase 2：App + 小组件

目标是把内容通过小组件呈现给用户，App 作为配置和数据源。

> 最低系统要求：iOS 17+ / macOS 14+

交付内容：

- iOS/macOS App（配置入口 + 后台数据源）
- 匿名 device_id
- 地点权限申请 + 天气 API 接入（透传后端，用户无感知）
- 后台拉取谕句/宜/忌，写入 App Group
- 小组件样式/主题配置与实时预览
- iOS/macOS 桌面小组件
- 小组件每日刷新（通过 WidgetKit Timeline 实现，刷新时间由系统调度，App 前台时主动调用 `reloadAllTimelines()` 作为兜底）
- 小组件点击交互（AppIntent）
- Supabase RLS 访问策略配置

阶段结果：

- App 能后台拉取每日内容并写入 App Group
- 小组件能稳定展示谕句/宜/忌，自动刷新和点击交互可用
- 天气和地点能作为筛选参数透传后端
- 单台设备尽量不重样
- 后台能看到基础分发效果

## 5. Phase 1 详细需求

### 5.1 输入类型

第一版建议先支持：

- txt
- epub

PDF 后面再加。第一阶段先把文本处理链路做稳。

### 5.2 内容来源类型

- 图书
- 歌词
- 文章 / 散文

第一版先按统一的"文本源"处理，只在来源类型上做区分。

### 5.3 内容库拆分

数据库中的正式内容拆成三类：

- 谕句库
- 宜库
- 忌库

三类内容独立存储，不绑定成固定组合。

### 5.4 第一版最小标签

第一版只保留这些字段：

- `language`
- `emotion`
- `weather_fit`
- `place_fit`

这几个先够用。后面要扩展，再补更多维度。

### 5.5 后台核心流程

1. 上传图书或文本
2. 系统自动抽取正文
3. 系统自动切片
4. AI 对片段做初筛
5. 生成候选谕句、候选宜、候选忌
6. 进入审核队列
7. 人工逐条审核
8. 审核通过后分别进入三类正式内容库

### 5.6 审核页要求

每条审核卡片至少展示：

- 原文片段
- 来源信息：书名、作者、语言、章节或位置
- AI 摘要
- 候选谕句
- 候选宜
- 候选忌
- 标签
- 分数或推荐等级

支持的审核动作：

- 入库
- 放弃
- 修改后入库
- 退回重跑

审核页重点是效率，不需要一开始就做复杂交互。

## 6. Phase 2 详细需求

### 6.1 App 核心功能

App 不直接展示内容，作为后台数据源和小组件配置入口：

- 首次启动生成匿名 `device_id` 并持久化
- 申请定位权限，获取用户地点
- 后台调用天气 API，获取当前天气（用户无感知）
- 直接查询 Supabase（anon key），以 `device_id`、`weather`、`place`、`emotion` 为筛选参数，排除 `delivery_logs` 中已看内容
- 将查询结果写入 App Group 共享存储
- 提供小组件样式/主题配置界面
- 提供小组件实时预览

### 6.2 小组件核心功能

- 读取 App Group 数据，展示谕句、宜、忌
- 定时刷新（WidgetKit Timeline）
- 点击交互（AppIntent，iOS 17+ Interactive Widgets）

### 6.3 设备级去重方案

没有用户账号，要支持每台设备自己的去重记录。

做法：

1. App 首次启动时生成 `device_id` 并持久化到本地
2. 将 `device_id` 注册写入 Supabase `devices` 表
3. 每次查询内容时，排除 `delivery_logs` 中该设备已看过的 `content_id`
4. 内容展示后，将记录写入 `delivery_logs`

这样可以做到：

- 用户 A 的内容尽量不和自己重复
- 用户 B 的内容也有自己的去重历史
- 不需要登录体系，全程匿名

### 6.4 第一版上下文输入

第一版先只用这些上下文做内容分发：

- `weather`（天气 API 获取，用户无感知）
- `place`（CoreLocation 获取）

步数、轨迹、连续几天状态这类更复杂的数据以后再加。

### 6.5 后续优化方向

- 投放统计
- 内容重复率分析
- 低质量内容下线
- 更好的内容组合策略

## 8. 技术方向

### 8.1 后端与数据库

采用 Supabase PostgreSQL 作为唯一数据库。

- 后台管理系统使用 service_role key，不受 RLS 限制
- App 使用 anon key，通过 RLS 只读已审核内容，并可写入自己的设备记录

### 8.2 数据库核心表

- `sources`
- `source_segments`
- `oracle_quotes`
- `daily_dos`
- `daily_donts`
- `devices`
- `delivery_logs`

### 8.3 分发原则

- 谕句、宜、忌分开筛选
- 按 `emotion / weather / place` 做基础匹配
- 查询时排除 `delivery_logs` 中该设备已看过的内容
